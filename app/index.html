Fase 2 · ArgoCD — La vera GitOps

Continuous Sync

ArgoCD monitora il repo GitHub e sincronizza automaticamente. Sync Policy = Automatic.

Sync Policy: Auto Self-healing

Rollback nativo

Cambi via Git commit: da 2→3 repliche, ArgoCD applica in cluster. Git è la single source of truth.

git commit -m "scale to 3 replicas"
→ ArgoCD rileva → deploy automatico

UI + port-forward

Accesso web alla dashboard ArgoCD con credenziali admin. Stato in tempo reale e diff visivo.

https://10.5.0.83:8080
🔁 Ogni modifica al repo Git → ArgoCD applica sul cluster → differenze segnalate → drift prevention.
Fase 3 · Prometheus + Grafana (Osservabilità)

kube-prometheus-stack

Installato con Helm: Prometheus (metriche), Grafana (dashboard), AlertManager, Node exporter.

Helm PromQL

Alert rule reale

PodCrashLooping — rileva container in crash nel namespace lab tramite PrometheusRule.

expr: rate(kube_pod_container_status_restarts_total[5m]) * 60 > 0
for: 1m → stato FIRING su Grafana

Dashboard predefinite

"Kubernetes / Compute Resources / Cluster" — uso CPU/RAM storico e live. Stesso approccio Zabbix ma automatico.

Node Exporter kube-state-metrics
Query PromQL essenziale: rate(container_cpu_usage_seconds_total{namespace="lab"}[5m])
Fase 4 · Pipeline CI/CD con GitHub Actions + GHCR

GitHub Actions workflow

Trigger su push in app/ → build immagine Docker → push su GHCR → aggiornamento automatico del tag nel deployment.yaml

name: Build and Push
docker/build-push-action@v5
sed -i "s|image:.*|image: ghcr.io/.../web:${{ github.sha }}|" deployment.yaml
git commit & push

Rolling update senza downtime

ArgoCD rileva il nuovo tag → k3s esegue rolling update → traffico zero-downtime.

RollingUpdate Immutable tag

App Custom HTML
